# 0.2.0 — The Agent Composition Graph

This release rebuilds the scanner's core around a first-class **composition
graph**. Instead of inferring component relationships from manifest path shapes,
OpenACA now constructs a real graph of the agent stack and derives scope,
lineage, and attribution from its edges — and exports that graph in the Agent
BOM.

## Highlights

- **The Agent Composition Graph is now the scanner's source of truth.**
  (ADR-0037, plan 033) OpenACA builds a graph by recursive descent over the
  target — nodes for plugins, skills, MCP servers, hooks, agents, commands, and
  per-occurrence package nodes; edges for containment and declaration. Component
  scope, lineage, and "which plugin introduced this" are derived from graph
  edges, replacing the old `_classify_dep_manifest` path heuristics. Repo and
  endpoint modes share the same descent, with boundary-aware skill/plugin
  traversal and lockfile-preferred dependency emission.

- **Attribution is now graph-derived, not a stored field.** (breaking for BOM
  consumers — see Compatibility) The stored `attributed_to` model field and the
  `openaca:attributed_to` BOM property are gone; parentage now lives in the
  graph. The Agent BOM encodes it as CycloneDX `dependencies[]` edges, each
  finding carries a full `component_path` lineage, and the inventory tree nests
  from graph edges. SARIF and the JSON finding output still surface an
  `attributed_to` field ("which plugin do I remediate?") — now computed from the
  graph's nearest-plugin-ancestor rather than read from a stored field.

- **Graph-native component identity.** (breaking) Package identity is no longer
  parent-qualified — a package resolves to one stable identity regardless of
  which plugin or skill pulled it in, and the relationship is expressed by graph
  edges instead of folded into the identity string. Skill artifact coordinates
  are now full skill-tree content hashes.

- **SkillSpector integrated as an external observation source.** OpenACA can run
  NVIDIA SkillSpector (`--scanner nvidia-skillspector`) and ingest its SARIF,
  classified by claim type — declarative supply-chain rules route to posture,
  behavioral rules to observations — with all 64 upstream rule IDs mapped and
  severities normalized (ADR-0033 / ADR-0034 for the SARIF-mapping and
  observation-source boundary). A requested-but-missing scanner binary now
  hard-fails the scan instead of silently continuing.

- **Skill-bundled dependency scanning (SC4 coverage).** OpenACA now scans
  dependency manifests bundled *inside* skills in both repo and endpoint modes,
  closing the gap where a vulnerable package vendored under a skill went
  undetected.

- **Graph-backed remote upload, redaction-hardened.** Remote sync emits
  graph-backed BOM edges and full finding `component_path`, uploads
  external-scanner findings, and surfaces scanner warnings to stderr. Redaction
  was hardened: absolute-path bom-refs, `file://` URIs, and Unix absolute paths
  embedded in Bash-filter evidence are stripped before upload, while embedded
  URLs are preserved and bom-ref uniqueness is kept for out-of-root plugins.

- **Scan progress for slow phases.** Long scans now report progress instead of
  appearing to hang.

- **Bug fixes.** A malformed plugin manifest no longer suppresses sibling
  components; gitignore is honored in nested repo descents and bundled
  plugin/skill surfaces; symlinked project-skill keys normalize by logical path;
  skill `allowed-tools` parsing follows the agentskills.io spec (space-separated
  entries and parenthesized filters), and `Bash(filter:*)` is detected as
  executable; plus numerous Codex-review fixes across `graph_build`, `scan`,
  `sarif`, and `bom_lint`.

## Install

`uv tool install openaca==0.2.0` or `pip install openaca==0.2.0`. Or:
`curl -fsSL https://openaca.dev/install.sh | sh`.

## Compatibility

First release since the public `0.1.0`, and it changes several output shapes
that downstream consumers will notice:

- **The BOM `openaca:attributed_to` property is removed** (along with the stored
  `attributed_to` model field). BOM consumers that read that property must
  instead follow the CycloneDX `dependencies[]` edges (parent `bom-ref` → child
  `bom-ref`) to recover "which plugin introduced this." **SARIF and the JSON
  finding output keep their `attributed_to` field** — now derived from the
  composition graph rather than a stored field, so its meaning is unchanged for
  downstream consumers. (`docs/openaca-bom-schema.md` and
  `docs/sarif-conventions.md` updated to match.)
- **Package identity is no longer parent-qualified** — the same package resolves
  to one identity regardless of the plugin/skill that pulled it in; the
  relationship is now expressed by graph edges. Identity strings that consumers
  matched on may change.
- **Skill artifact coordinates** are now full skill-tree content hashes.
- **A requested external scanner that is missing now aborts the scan** (it
  previously warned and continued).

The Agent BOM format and component identities remain pre-1.0 and may still
evolve.
